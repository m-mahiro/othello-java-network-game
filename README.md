## 開発ルール等
- 1つの意味の塊を持つコードは空改行を入れずに塊にして、その先頭行の上にコメントを入れる。
- いくつかのコードの塊を横断して意味を持つコメントの場合、`============ コメント =========`という形式をとる。
- それぞれのクラスの責任範囲をちゃんと意識して開発する。
- そのクラスが利用される範囲をちゃんと考えて、アクセス制御をする。
- 依存関係のループをなくす
- 追加機能がある際に、変更範囲が最小限で済むようにする。
- 開発はIssue駆動でおこない、ブランチは`issue<Issue番号>とする`
- mainブランチに直接pushしない。(GitTreeがかっこよくなるから)
- なるべく一つのコミットに無関係な変更を混ぜない
- なるべくIssueに関係ない変更を混ぜない（途中で気づいたことをその時にやってしまうとスタックオーバーフローするから）
- 開発中に学んだことは忘れないように、Issue等にちゃんとメモしておく。
- ハードコーディングを避けるためだけの定数は大文字
- 変更する予定のないオブジェクトはfinalにする。
  - 参照型も、再代入する必要がなければfinalにする
  - 参照型はなるべく再代入しないようにする（そのオブジェクトを取り出した後遠くの方で覚えていることがもしあった時にバグる）
- パブリックメソッド、プロテクティッドメソッド、プライベートメソッドはこの順に固める。

## アノテーションコメント
自分を使用せずに、かならずメモする。必要に応じてGitHubでIssueを立てる
- TODO: 自分のためのタスク管理
- REVIEW: 後で考え直す必要がある
- NOTE: その実装にした理由
- SMELL: アンチパターンに一致
- FIXME: バグを発見
- HACK: その場しのぎのコード
